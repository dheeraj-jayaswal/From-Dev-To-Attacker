# Day 17 — robots.txt & sitemap.xml: The Map the Company Handed Over Voluntarily

**Severity:** Medium → High (depends on what's exposed) | **Category:** Recon & Fingerprinting | **Series:** #FromDevToAttacker

---

## 🔍 What It Is

`robots.txt` tells search engine crawlers which paths *not* to index — and since it's an honor-system protocol with zero enforcement, every `Disallow:` entry is effectively a public list of paths the developer considered sensitive enough to hide from Google, handed to anyone who simply requests the file. `sitemap.xml` does the opposite — it's a complete, structured inventory of every URL the site wants indexed, often including API routes and admin sections that were never meant to be treated as a target list.

## 🌱 Why It Exists

Developers write `Disallow:` entries thinking only about search engine behavior — the intent is purely "don't show this in Google results," with no thought given to the fact that the file itself is unauthenticated and publicly readable by anyone, crawler or not. Sitemaps have the same blind spot in reverse: they're generated to help indexing, often auto-generated by a CMS or framework that includes every route it knows about without distinguishing public-facing pages from internal-only ones.

## ⚔️ How To Find It

```bash
curl -s https://target.com/robots.txt | grep -i disallow

curl -s https://target.com/sitemap.xml | grep -oE '<loc>[^<]+</loc>'
```

The detail worth building into your process every time: sort sitemap entries by their `<lastmod>` date. Recently modified pages correlate strongly with recently shipped features — which means less time for a security review to have caught anything, and a higher hit rate for genuinely new (and less hardened) functionality.

## 💥 How To Exploit It

Test every `Disallow:` path directly rather than assuming it's protected — a surprising number of these paths return a full 200 with no authentication at all, since the developer's mental model was "crawlers won't find it" rather than "this needs an access control check." Follow sitemap index files recursively; large applications frequently split sitemaps by section, and an `sitemap-admin.xml` or `sitemap-api.xml` entry is exactly the kind of thing worth chasing down first.

## 🌍 Where This Actually Bites

In a **banking** client's public site, `robots.txt` listed `/api/internal/` and `/staging/` as disallowed paths — both resolved with a full 200 response and no authentication, handing over an internal API surface and a staging environment running an older, unpatched build of the same application. In **education** platforms, sitemap entries sorted by `lastmod` have led me directly to a newly-launched admissions portal section still carrying default test data and a debug flag left on from the deployment — found purely because it was the most recently modified URL in an otherwise routine sitemap.

## 📋 How To Report It

Grade by what testing each disallowed/sitemap-listed path actually returns — a `Disallow: /admin/` entry that correctly requires authentication is a Low, informational "reduce fingerprinting" note at most; the same entry returning an unauthenticated 200 is High or Critical depending on what's behind it. Frame the remediation precisely: `robots.txt` is not an access control mechanism, and any path listed there specifically because it's sensitive needs a real authentication check independent of whether it's crawled.
