# Day 23 — Exposed S3 Buckets: Public Read on a Filing Cabinet Nobody Locked

**Severity:** High → Critical (based on contents) | **Category:** Sensitive Data Exposure | **Series:** #FromDevToAttacker

---

## 🔍 What It Is

An S3 bucket set to public-read allows anyone to list and download its contents via a simple HTTP request — no AWS account or credentials required. Bucket names are globally unique and highly guessable (`company-backup`, `company-prod`, `company-uploads`), which means finding a candidate bucket is often a matter of pattern-guessing rather than genuine discovery.

## 🌱 Why It Exists

This is almost always a policy misconfiguration made early in a project's life, when a bucket is set to public for a legitimate, narrow reason — serving static assets to a website, for instance — and that same public policy is later reused or left in place when the bucket's purpose shifts to something sensitive, like backups or data exports. Nobody revisits bucket ACLs as a matter of routine the way they'd revisit, say, an API's authentication config, because storage buckets don't feel like "the application" to most engineering teams.

## ⚔️ How To Find It

```bash
# Test common naming patterns for the target
for suffix in "" -prod -backup -data -uploads -staging; do
  code=$(curl -s -o /dev/null -w "%{http_code}" "https://target${suffix}.s3.amazonaws.com")
  echo "$code target${suffix}"
done

# Once found, list without any AWS credentials at all
aws s3 ls s3://target-backup --no-sign-request
```

`s3scanner` automates the pattern sweep across a larger candidate list efficiently, and GrayHatWarfare is worth a quick check too — plenty of public buckets are already indexed there from prior discovery by others, meaning zero active scanning required on your part.

## 💥 How To Exploit It

Once listing access is confirmed, the priority is triage, not blanket downloading — filter the file listing for extensions and names that signal high sensitivity (`.sql`, `.env`, `.pem`, filenames containing "export" or "backup") before pulling anything, and pull the minimum needed to prove the finding rather than the entire bucket.

## 🌍 Where This Actually Bites

In a **banking**-adjacent fintech engagement, a public bucket named after the company plus `-backup` contained a full nightly database export — unencrypted, containing customer account data, sitting in a bucket that had clearly been set to public years earlier for an unrelated static-asset purpose and never revisited when its role changed to backups. In **retail/ecommerce**, I've found public buckets holding customer data exports (order history, addresses) generated by a marketing analytics integration — the bucket's public policy was correct for the original use case (serving product images), but a second, more sensitive workload was later pointed at the same bucket without anyone re-evaluating the access policy for the new content.

## 📋 How To Report It

Grade based on file sensitivity, not on the existence of public access alone — a bucket serving public marketing images correctly configured as public-read isn't a finding at all, while the same access level on a bucket containing database dumps or PII exports is Critical. Include the bucket name, confirmation of public-read access (a single `aws s3 ls --no-sign-request` output line is sufficient evidence), and a short, specific list of the most sensitive file types found — not a full directory dump — plus a clear recommendation to enable account-level Block Public Access as the systemic fix, not just correcting the one bucket's policy.
